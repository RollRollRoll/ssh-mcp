## 一页纸概览

- **一句话定义**：SSH MCP 是一个通过 MCP `stdio` Transport 暴露 SSH 命令、交互终端和文件传输能力的本地服务，供支持工具调用及 form elicitation 的 MCP 客户端使用（`README.md:1-25`）。
- **核心价值**：让 AI 客户端可以操作预先登记的开发或测试主机，同时把主机、认证方式、文件根目录和低风险命令 Profile 锁定在启动配置中（`README.md:27-31`）。
- **架构风格**：单进程 MCP Server；客户端经标准输入输出调用工具，服务通过 `ssh2` 连接 Linux 或 Windows OpenSSH Server（`src/server.ts:2-27`）。
- **主要技术栈**：Node.js 24、TypeScript、MCP SDK、ssh2、Zod、YAML、Vitest（`package.json:6-31`）。
- **运行形态**：长驻 `stdio` 子进程，不提供 HTTP 服务，也不是独立 SSH CLI（`README.md:11-19`）。
- **代码规模感**：单仓单包，`src/` 下约 8,662 行 TypeScript，按配置、审批、SSH、命令、会话、传输等职责分模块。

```text
支持 MCP 的客户端
      │ stdio JSON-RPC
      ▼
SSH MCP Server
      ├── 配置、审批与策略
      ├── 命令、PTY 会话与操作状态
      └── 路径守卫与 SFTP 传输
                 │ SSH/SFTP
                 ▼
        已登记的开发/测试主机
```

### 最短使用路径

1. 安装 Node.js 24，并准备 SSH Agent、Pageant 或无口令私钥文件。
2. 执行 `npm ci && npm run build`。
3. 参照 [配置说明](./docs/configuration.md) 创建一份 YAML 配置。
4. 把 `node /绝对路径/ssh-mcp/dist/index.js --config /绝对路径/ssh-mcp.yml` 注册为 MCP 客户端的 `stdio` Server。
5. 在客户端先调用 `hosts_list`，再调用命令、会话或文件工具。
6. 首次连接确认主机指纹；有副作用的操作按客户端表单进行一次性审批。

## 1. 项目定位与功能

### 1.1 定位与核心价值

项目面向希望让 MCP 客户端操作受控 SSH 环境的开发者。它不接受动态主机和密码，调用方只能选择配置文件中已经登记的主机、认证方式和路径范围。生产主机、多人权限、长期审计、HTTP/UI、端口转发和自动重试明确不在当前版本范围内（`README.md:44-46`）。

### 1.2 功能地图

| 功能 | MCP 工具 | 输入摘要 | 输出摘要 | 源码位置 |
|---|---|---|---|---|
| 查看主机 | `hosts_list` | 无参数 | 主机别名、环境、平台、Shell、连接状态 | `src/tools/hosts-list.ts:5-27` |
| 执行原始命令 | `command_run` | 主机列表、命令、并行/顺序模式 | `operationId`、运行状态 | `src/tools/command-run.ts:11-88` |
| 执行低风险 Profile | `profile_run` | Profile ID、主机、参数、执行模式 | `operationId`、运行状态 | `src/tools/profile-run.ts:12-89` |
| 交互终端 | `session_open/write/read/resize/close` | 会话、终端尺寸、输入字节或游标 | 会话状态及 PTY 输出帧 | `src/tools/session-tools.ts:12-166` |
| 文件上传/下载 | `file_upload/file_download` | 主机、源/目标、递归、覆盖、执行模式 | `operationId`、运行状态 | `src/tools/file-transfer-tools.ts:12-120` |
| 查询/取消长操作 | `operation_get/operation_cancel` | 操作 ID、输出游标、最大字节数 | 状态、输出帧、结果或错误 | `src/tools/operation-control.ts:5-59` |

### 1.3 主要输入与输出

- **输入**：客户端通过 MCP JSON-RPC 提交结构化工具参数；服务只从启动参数或 `SSH_MCP_CONFIG` 定位 YAML 配置（`src/server.ts:31-48`）。
- **输出**：工具返回结构化结果；命令与传输先返回 `operationId`，随后用 `operation_get` 分页读取状态和输出（`src/tools/operation-control.ts:28-53`）。
- **日志**：标准输出只用于 MCP 帧，诊断日志写标准错误（`README.md:19`）。

## 2. 整体架构

### 2.1 架构风格

项目是单进程、模块化的 MCP Server。`startServer` 加载并冻结配置，创建主机注册表、操作管理器、审批服务、SSH 适配器、会话管理器和传输服务，然后注册工具并连接 `StdioServerTransport`（`src/server.ts:82-222`）。模块间通过进程内函数调用协作，没有数据库、缓存或消息队列。

### 2.2 主要模块

| 模块 | 路径 | 职责 |
|---|---|---|
| 启动与装配 | `src/index.ts`、`src/server.ts` | 启动服务、装配依赖、注册工具、优雅关闭 |
| 配置 | `src/config/` | 读取 YAML、校验字段、禁止别名/标签、深冻结 |
| 审批与策略 | `src/approval/`、`src/policy/` | 生成不可变操作意图、请求表单审批、匹配低风险 Profile |
| SSH 与信任 | `src/ssh/` | SSH 连接、TOFU 指纹确认、信任存储、平台探测 |
| 命令与会话 | `src/commands/`、`src/sessions/` | 执行命令、管理 PTY 会话及输入输出 |
| 长操作 | `src/operations/`、`src/multihost/` | 状态机、输出缓冲、多主机并发/顺序协调 |
| 文件传输 | `src/transfers/`、`src/paths/` | SFTP 文件/目录传输、原子目标、路径和链接边界检查 |
| MCP 工具 | `src/tools/` | 12 个公开工具的输入输出 Schema 与 Handler |

### 2.3 数据流与控制流

- **数据流**：MCP 工具参数 → Zod 校验 → 主机/路径/策略校验 → 审批或 Profile 匹配 → SSH/SFTP 操作 → 操作缓冲区 → MCP 结构化结果。
- **控制流**：服务常驻等待 MCP 请求；长操作异步执行并立即返回 `operationId`。客户端用游标轮询输出，或请求取消（`src/commands/command-runner.ts:69-99`、`src/tools/operation-control.ts:48-59`）。
- **失败处理**：项目不自动重试。超时、连接中断或无法确认远端结果时可进入 `unknown`，以避免把不确定副作用错误地声明为失败或成功（`docs/configuration.md:73-77`）。

### 2.4 外部系统

- **MCP 客户端**：必须支持 MCP `2025-11-25`、`stdio`、工具调用和 form elicitation，才能使用全部能力（`README.md:21-25`）。
- **SSH 主机**：仅支持 YAML 中登记的 Linux 或 Windows 开发/测试主机（`src/config/schema.ts:45-64`）。
- **本地凭据设施**：SSH Agent、Pageant 或私钥文件，三选一（`src/config/schema.ts:15-33`）。

## 3. 技术栈

- **运行时**：Node.js `>=24 <25`，ES Module（`package.json:5-8`）。
- **语言**：TypeScript，目标 ES2024，启用严格模式（`tsconfig.json:2-13`）。
- **MCP**：`@modelcontextprotocol/sdk`，使用 `McpServer` 和 `StdioServerTransport`（`src/server.ts:2-4`）。
- **SSH/SFTP**：`ssh2`（`package.json:20-24`）。
- **配置与校验**：`yaml` 解析 YAML 1.2，`zod` 校验配置及工具输入输出（`src/config/loader.ts:41-60`）。
- **测试**：Vitest；测试分为 unit、contract、integration 和 acceptance（`package.json:9-18`）。
- **数据存储**：不适用；只在本地 `trustStore` 文件中保存用户确认过的主机公钥指纹（`docs/configuration.md:69-71`）。

## 4. 代码结构与核心实现

### 4.1 目录地图

```text
ssh-mcp/
├── src/                 # 服务实现
│   ├── tools/           # 12 个 MCP 工具
│   ├── approval/        # 一次性审批与操作意图
│   ├── ssh/             # SSH、TOFU 和信任存储
│   ├── commands/        # 命令构造与运行
│   ├── sessions/        # PTY 会话
│   ├── transfers/       # SFTP 文件/目录传输
│   ├── paths/           # Linux/Windows 路径边界
│   └── operations/      # 长操作状态和输出缓冲
├── tests/               # 单元、契约、集成和验收测试
├── docs/                # 配置、规格、设计与任务文档
├── scripts/             # 集成测试入口
└── dist/                # TypeScript 构建产物
```

目录按技术职责划分。要查看调用参数先读 `src/tools/`，要查看安全边界读 `src/approval/`、`src/policy/`、`src/paths/` 和 `src/ssh/`。

### 4.2 程序入口与启动流程

1. `node dist/index.js` 进入 `runMain`，并注册 SIGINT/SIGTERM 关闭逻辑（`src/index.ts:5-19`）。
2. `startServer` 解析 `--config` 或 `SSH_MCP_CONFIG`，要求绝对路径（`src/server.ts:31-48`）。
3. `ConfigLoader` 读取 YAML，拒绝锚点、别名和自定义标签，并深冻结配置（`src/config/loader.ts:21-60`、`src/config/loader.ts:69-99`）。
4. 创建操作、会话、审批、SSH、策略和传输组件（`src/server.ts:96-217`）。
5. 连接 `StdioServerTransport`，开始接收 MCP 请求（`src/server.ts:219-222`）。

### 4.3 核心配置模型

- 一份配置最多登记 10 台主机，每台必须声明环境、平台、地址、账号、认证、Shell 和远端根目录（`src/config/schema.ts:45-64`、`src/config/schema.ts:150-165`）。
- 配置环境只能是 `development` 或 `test`，明确排除了生产环境（`src/config/schema.ts:47`）。
- Linux 强制使用 POSIX Shell，Windows 强制使用 PowerShell（`src/config/schema.ts:55-64`）。
- `lowRiskProfiles` 只能引用登记主机，Profile ID、主机别名和参数名均需唯一（`src/config/schema.ts:97-117`、`src/config/schema.ts:166-195`）。

### 4.4 核心实现

项目的核心安全设计是“审批内容与实际执行内容严格相等”。`command_run` 先生成带摘要的操作意图，审批后再次比较 `kind`、`digest` 和规范化 JSON；不一致就返回 `APPROVAL_INTENT_MISMATCH`（`src/tools/command-run.ts:56-85`、`src/tools/command-run.ts:93-95`）。

命令不会跨平台翻译：Linux 使用配置的 Shell 加 `-lc`，Windows 使用 PowerShell `EncodedCommand`，原始命令语义由调用方负责（`src/commands/command-builder.ts:3-17`）。

## 5. 端到端代表性流程

### 流程 A：执行一条远端命令

**触发**：客户端调用 `command_run`。

```text
[输入校验] src/tools/command-run.ts:11-15
   ↓ 解析全部主机并生成不可变 OperationIntent
[请求审批] src/tools/command-run.ts:50-64
   ↓ 审批内容与执行内容再次精确比对
[启动后台操作] src/tools/command-run.ts:65-80
   ↓ 构造平台原生命令并建立 SSH 连接
[执行和采集输出] src/commands/command-runner.ts:101-172
   ↓
[返回 operationId；客户端轮询状态和输出]
src/tools/operation-control.ts:48-59
```

关键点：审批前不会执行命令；单主机和多主机都返回异步操作 ID；项目不自动重试。取消时会尽力发送 TERM，无法确认远端最终状态时可能返回 `unknown`（`src/commands/command-runner.ts:79-99`、`src/commands/command-runner.ts:175-209`）。

### 流程 B：上传文件或目录

**触发**：客户端调用 `file_upload`。

```text
[输入校验] src/tools/file-transfer-tools.ts:12-22
   ↓ 审批前只做词法路径边界判断
[本地/远端根目录校验] src/tools/file-transfer-tools.ts:69-85
   ↓ 生成包含精确路径、recursive、overwrite 的审批意图
[一次性审批] src/tools/file-transfer-tools.ts:87-100
   ↓ 再次验证已批准意图并启动 SFTP
[后台传输] src/server.ts:205-217
   ↓
[operation_get 查询进度和最终状态]
```

关键点：本地路径必须位于 `localRoots`，远端路径必须位于对应主机的 `remoteRoots`。目录操作拒绝链接、重解析点和树内挂载点（`docs/configuration.md:73-77`）。

## 6. 配置、依赖与运行

### 6.1 前置条件

- Node.js 24 LTS。
- 可用的 SSH Agent、Pageant 或本地私钥文件。
- 一台或多台仅用于开发/测试的 SSH 主机。
- 支持 `stdio`、MCP Tools 和 form elicitation 的 MCP 客户端。客户端如果不支持表单，只能安全使用 `hosts_list`；副作用工具会返回 `APPROVAL_UNSUPPORTED`（`README.md:21-25`）。

### 6.2 安装与构建

在项目目录执行：

```sh
cd /path/to/ssh-mcp
npm ci
npm run build
```

构建产物入口为：

```text
/path/to/ssh-mcp/dist/index.js
```

### 6.3 创建配置

建议先为配置和信任文件准备独立目录。以下是最小 Linux 示例，所有路径和主机信息都需要替换为你自己的**绝对路径**：

```yaml
version: 1
trustStore: /absolute/state/ssh-mcp/trust.json
localRoots:
  - /absolute/local/workspace
hosts:
  - alias: linux-test
    environment: test
    platform: linux
    host: 192.0.2.10
    port: 22
    username: developer
    auth:
      type: agent
      socket: /absolute/path/to/ssh-agent.sock
    shell:
      type: posix
      command: /bin/sh
    remoteRoots:
      - /absolute/remote/workspace
```

认证方式三选一：

```yaml
# SSH Agent
auth:
  type: agent
  socket: /absolute/path/to/agent.sock

# Pageant
auth:
  type: pageant

# 本地私钥文件；不支持带口令私钥
auth:
  type: privateKeyFile
  path: /absolute/path/to/id_ed25519
```

完整字段、Windows 示例、低风险 Profile 和预算默认值见 [docs/configuration.md](./docs/configuration.md)。不要把密码、私钥正文、口令或令牌写进 YAML（`docs/configuration.md:59-67`）。

### 6.4 接入 MCP 客户端

不同客户端的配置文件位置和外层字段名称不同，但 `stdio` Server 的共同参数如下：

```json
{
  "command": "node",
  "args": [
    "/path/to/ssh-mcp/dist/index.js",
    "--config",
    "/absolute/path/to/ssh-mcp.yml"
  ]
}
```

也可以通过环境变量传入配置路径：

```json
{
  "command": "node",
  "args": [
    "/path/to/ssh-mcp/dist/index.js"
  ],
  "env": {
    "SSH_MCP_CONFIG": "/absolute/path/to/ssh-mcp.yml"
  }
}
```

`--config` 和 `SSH_MCP_CONFIG` 都必须是绝对路径；存在命令行参数时只接受严格的 `--config <absolute-path>` 形式（`src/server.ts:31-48`）。

### 6.5 首次使用

连接 MCP Server 后，可以按以下顺序操作：

1. 调用 `hosts_list`，确认主机别名已加载。
2. 调用 `command_run`，例如让客户端“在 `linux-test` 上执行 `pwd`”。
3. 客户端展示完整命令和摘要后，确认一次性审批。
4. 首次 SSH 连接时，核对并确认服务器公钥算法和指纹。指纹会写入 `trustStore`；以后发生变化会直接拒绝（`docs/configuration.md:69-71`）。
5. 工具返回 `operationId` 后，调用 `operation_get` 读取 stdout、stderr 和最终状态。

对应的结构化调用等价于：

```json
{
  "hosts": ["linux-test"],
  "command": "pwd",
  "executionMode": "parallel"
}
```

查询结果：

```json
{
  "operationId": "由 command_run 返回的 ID",
  "cursor": 0,
  "maxBytes": 65536
}
```

### 6.6 常用工具参数

#### 执行原始命令

```json
{
  "hosts": ["linux-test"],
  "command": "git status --short",
  "executionMode": "parallel"
}
```

`executionMode` 可为 `parallel` 或 `sequential`，省略时命令默认并行（`src/tools/command-run.ts:11-15`、`src/tools/command-run.ts:56-61`）。

#### 上传文件

```json
{
  "hosts": ["linux-test"],
  "localSource": "/absolute/local/workspace/app.tar.gz",
  "remoteTarget": "/absolute/remote/workspace/app.tar.gz",
  "recursive": false,
  "overwrite": false,
  "executionMode": "parallel"
}
```

#### 下载目录

```json
{
  "hosts": ["linux-test"],
  "remoteSource": "/absolute/remote/workspace/logs",
  "localTarget": "/absolute/local/workspace/downloads",
  "recursive": true,
  "overwrite": false,
  "executionMode": "parallel"
}
```

#### 打开交互会话

```json
{
  "host": "linux-test",
  "columns": 120,
  "rows": 30
}
```

得到 `sessionId` 后，用 `session_write` 发送输入，用 `session_read` 按游标读取输出，结束时调用 `session_close`（`src/tools/session-tools.ts:73-166`）。每次 `session_write` 都会单独审批（`src/tools/session-tools.ts:91-119`）。

### 6.7 低风险 Profile

Profile 在 YAML 中预先定义固定可执行文件、固定参数和受限动态参数。匹配成功后，`profile_run` 不走普通原始命令审批流程；任何 Profile ID、主机或参数不完全匹配都会在连接前拒绝（`src/tools/profile-run.ts:43-89`）。

示例定义：

```yaml
lowRiskProfiles:
  - id: list-build-output
    hostAliases: [linux-test]
    platform: linux
    executable: /usr/bin/find
    fixedArgs: [/absolute/remote/workspace/build, -maxdepth, "1", -type, f]
    parameters: []
```

调用参数：

```json
{
  "profileId": "list-build-output",
  "hosts": ["linux-test"],
  "parameters": {},
  "executionMode": "parallel"
}
```

### 6.8 验证与部署

项目提供以下验证命令（`README.md:33-42`）：

```sh
npm run check
npm run test:contract
npm run test:integration:linux
npm run test:acceptance
```

Windows 集成测试需要受控 Windows OpenSSH Server。仓库有 GitHub Actions，Linux 和 Windows 都使用 Node.js 24 执行构建、契约、集成和验收测试（`.github/workflows/ci.yml:1-36`）。项目没有 Dockerfile 或独立部署清单；正常使用方式是由 MCP 客户端把它作为本地子进程启动。

### 6.9 常见问题

- **启动时报缺少配置**：确认提供了 `--config` 或 `SSH_MCP_CONFIG`，且路径为绝对路径。
- **副作用工具返回 `APPROVAL_UNSUPPORTED`**：客户端不支持 form elicitation；换用满足 README 能力基线的客户端。
- **首次连接没有执行命令**：需要先完成 TOFU 主机指纹确认。
- **出现主机密钥变化错误**：先在 SSH 服务端和可信渠道核查，不要直接删除信任记录绕过。
- **文件传输被拒绝**：检查本地路径是否在 `localRoots`、远端路径是否在对应 `remoteRoots`，以及源是否包含符号链接、重解析点或挂载点。
- **操作一直不是 completed**：用 `operation_get` 查看 `state` 和 `error`；`unknown` 表示服务无法证明远端最终结果，不应直接重试可能产生副作用的操作。

## 7. 项目类型专项分析

### 7.1 MCP Server

- **判断依据**：依赖 `@modelcontextprotocol/sdk`，创建 `McpServer`，并使用 `StdioServerTransport`（`package.json:20-24`、`src/server.ts:2-4`）。
- **能力清单**：固定注册 12 个 Tools；未发现 Resources 或 Prompts（`README.md:25`、`src/server.ts:272-316`）。
- **Transport**：仅 `stdio`，无 SSE 或 HTTP（`src/server.ts:219`）。
- **Schema**：工具输入输出均由 Zod 严格校验，主要位于 `src/tools/*.ts`。
- **权限边界**：只允许登记主机；路径受本地/远端根目录约束；副作用操作要求审批；原始命令的执行范围不做语义白名单，但会完整展示并绑定审批内容。
- **外部连接**：只连接 SSH/SFTP 主机，不依赖数据库或云 API。

### 7.2 开发运维工具

- **标准流程**：配置受控主机 → MCP 客户端启动服务 → 查看主机 → 审批并执行操作 → 查询异步结果。
- **产物文件**：构建生成 `dist/`；运行时可能创建/更新 `trustStore`；下载操作会在 `localRoots` 中写文件。
- **Git/CI 关系**：服务本身不调用 Git 或 CI；客户端可通过 SSH 命令间接操作远端开发目录。仓库自身使用 GitHub Actions 验证 Linux/Windows 行为。
- **修改范围**：命令、PTY 和传输均可能产生远端或本地副作用，因此依赖审批、配置根目录和明确的副作用状态进行约束。

## 8. 综合评价

- **项目亮点**：审批不是简单的“点过确认就继续”，而是对批准前后的规范化操作意图做摘要和精确比对，降低审批后参数被替换的风险（`src/tools/command-run.ts:56-85`）。
- **优点**：配置严格、平台边界明确、长操作可查询与取消、Linux/Windows 都有集成测试，并且对无法确认的远端结果使用 `unknown` 而非虚假成功/失败。
- **局限**：客户端必须支持较新的 MCP form elicitation；只支持开发和测试环境；不支持密码或带口令私钥；无自动重试、断点续传、跨重启恢复和长期审计（`README.md:21-25`、`README.md:44-46`）。
- **适合场景**：本地 AI 开发助手操作个人或团队受控的开发机、测试机和 CI 验证机。
- **不适合场景**：生产变更、多人共享权限、强审计合规、堡垒机替代、需要复杂网络转发或凭据托管的环境。
- **二次开发建议**：代码模块边界和测试分层较清晰，适合在现有 12 个工具、安全意图和操作状态机之上扩展；新增副作用能力时应复用审批和 `unknown` 语义，避免绕开现有边界。
